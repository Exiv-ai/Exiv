# Exiv Strategic Battle Plan

> **"Summer Wars Protocol"**
> 巨大システムに対して、一人の開発者が知恵と共感で立ち向かう。

---

## 1. 戦略的前提

### 1.1 現在の戦場

2026年2月、AIエージェント市場は転換点にある。

- AIエージェントのセキュリティ脆弱性がサイバーセキュリティ研究者から繰り返し指摘されている
- 「ビッグテックにデータを渡したくない」という感情が世界的に高まっている
- ローカルLLMコミュニティ (r/LocalLLaMA等) が急成長し、自己主権的AIへの需要が顕在化

### 1.2 ウィンドウ

AIセキュリティへの関心が高まっている今こそ、
Exivの名前と思想を世界に刻む。

### 1.3 勝利の定義

4ヶ月で勝てない戦いと、勝てる戦いを明確に区別する。

**勝てない戦い（追わない）：**
- 機能数でOpenClawを上回る
- ユーザー数でOpenClawを超える
- 完成度の高い完全なプロダクトを出す

**勝てる戦い（ここに全力を注ぐ）：**

| 目標 | 達成条件 | 指標 |
|------|----------|------|
| 認知の獲得 | AI agent securityの文脈でExivが言及される | HN/Reddit投稿のリアクション |
| 技術的信頼 | 「Rust製・セキュリティファースト」が事実として認知される | GitHub Stars 1,000+ |
| コミュニティの芽 | ビジョンに共感した開発者が集まり始める | 初期コントリビューター 5-10人 |

## 2. ナラティブ：なぜExivが存在するのか

### 2.1 敵は「人」ではなく「問題」

OpenClawを名指しで攻撃しない。問題そのものを敵にする。

> **「AIエージェントは便利になった。だが今、あなたのデータもPC権限も丸見えだ。」**

OpenClawはその問題の象徴として自然に位置づく。
攻撃ではなく、問題提起。Exivはその回答。

### 2.2 サマーウォーズの構図

| 物語の要素 | 現実の対応 |
|-----------|-----------|
| OZ — 便利だが脆弱な巨大システム | 現行AIエージェント群のセキュリティ問題 |
| ラブマシーン — 制御を奪う脅威 | ユーザーデータ・権限の無秩序な露出 |
| 陣内家 — 普通の人々が知恵で立ち向かう | 日本人個人開発者 + オープンソースコミュニティ |
| アカウントを貸す世界中の人々 | プラグインを書き、コンテナを作るコントリビューター |
| 「よろしくお願いしまぁぁす！！」 | Exivの初回リリースとコミュニティ形成 |

この構図は **作り物ではなく、現実にそうなっている** ことが最大の強み。

### 2.3 核心メッセージ（30秒で伝わること）

> **「自分だけのAIパートナーを、自分のPCで。データは外に出ない。権限はあなたが握る。」**

技術的裏付け：
- **Rust** — パフォーマンスとメモリ安全性
- **サンドボックス** — プラグインごとのプロセス分離
- **パーミッション分離** — 能力注入モデル (Capability Injection)
- **SafeHttpClient** — DNS Rebinding対策を含むネットワーク制限

これらは「物語」ではなく「事実」。嘘をつく必要がない。

## 3. 超短期決戦タイムライン

### Month 1: 「動いているExiv」を見せる

**目標**: 30秒デモ動画の完成

- [ ] コアランタイムが起動し、ダッシュボードが表示される
- [ ] DeepSeek プラグイン経由でAIと会話できる
- [ ] 簡易な人格定義（AIコンテナのプロトタイプ）が動作する
- [ ] 30秒デモ動画を撮影する：インストール → コンテナ選択 → 会話

**最大のリスク**: 内部で完璧を追求し、何も外に出さないまま時間が過ぎる。
動く最小限を出すことが、完璧な設計書より価値がある。

### Month 2: 世界に投げる

**目標**: 最初の外部フィードバックとStars獲得

- [ ] Hacker Newsへの投稿（切り口：Rust × Security × AI Agent）
- [ ] r/LocalLLaMAへの投稿（切り口：ローカルAIパートナーの構築）
- [ ] r/rustへの投稿（切り口：Rustで書かれたAIコンテナプラットフォーム）
- [ ] ランディングページの公開（GitHub Pages or site/）
- [ ] セキュリティ比較記事（OpenClawとの構造的差異を技術的に解説）

**投稿戦略**: 1回の大きな投稿ではなく、異なる切り口で複数コミュニティに分散。
各コミュニティの関心軸に合わせてメッセージを調整する。

### Month 3: 仲間を集める

**目標**: 最初のPRがマージされる

- [ ] プラグインSDKドキュメントの整備
- [ ] 「最初のプラグインを書く」チュートリアル
- [ ] Good First Issue の設計と公開
- [ ] Discordコミュニティの開設
- [ ] 外部コントリビューターからの最初のPR

**入口の設計が全て**。サマーウォーズで世界中の人がアカウントを貸した場面 —
あれは「自分にもできることがある」という感覚。
プラグインSDKとCONTRIBUTING.mdがその入口になる。

### Month 4: 体験を届ける

**目標**: 「インストールして即使える」体験

- [ ] AIコンテナのプリセット配布（デモ用コンテナ 1-2個）
- [ ] ワンコマンドインストーラーの完成（install.sh / install.ps1）
- [ ] AIコンテナマーケットプレイスの構想公開
- [ ] OpenClawからの移行ガイド（セキュリティ比較を前面に）

| **英語圏へのリーチ** | 技術ドキュメントは英語。投稿も英語。コードは万国共通 |

## 5. 最大のリスクと対策

### リスク：早すぎる完璧主義

4ヶ月で負けるパターンは一つだけ。

> **内部で完璧を追求し、何も外に出さないまま時間が過ぎる。**

### 対策：「出す」を最優先にする判断基準

機能の追加・改善を検討する際、常にこの問いを立てる：

> **「これは、世界にExivを見せるために必要か？」**

Yesなら進める。Noなら後回し。

## 6. 指針

### 「正しく作る」より「正しく届ける」

アーキテクチャは既に正しい方向にある。
足りないのは、それを **世界に見せること** だけ。

### 「攻撃」ではなく「提案」

OpenClawを攻撃しない。問題を指摘し、回答を提示する。
敬意ある競争は、コミュニティからの信頼を生む。

### 「一人で勝つ」のではなく「仲間と勝つ」

サマーウォーズの勝利は一人の天才によるものではなかった。
世界中の人々が「自分にもできることがある」と感じて参加した結果だった。
Exivもそうあるべき。
