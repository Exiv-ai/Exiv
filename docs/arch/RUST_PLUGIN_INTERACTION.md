# VERS SYSTEM: Rust Plugin Interaction Mechanism

VERS SYSTEM における Rust 製ネイティブ・プラグイン（例：DeepSeek プラグイン）が Kernel（Core）とやり取りする仕組みを解説します。Rust プラグインは、システムと同じメモリ空間で動作し、最高速度の連携を実現します。

---

## 1. 3 つの連携レイヤー

Rust プラグインは、以下の 3 つの異なる方式を組み合わせて Kernel と通信します。

### ① トレイトによる直接対話 (Trait-based Interaction)
プラグインは Core が定義したインターフェース（`ReasoningEngine`, `MemoryProvider` 等）を直接実装します。
- **仕組み**: Kernel がプラグインのメソッド（例：`think()`）を直接呼び出します。
- **メリット**: データのコピーやシリアライズが発生せず、オーバーヘッドがゼロです。
- **用途**: 同期的な処理や、ミリ秒単位のレスポンスが求められる基幹機能。

### ② イベントバスによる非同期連携 (Event-First Communication)
全てのプラグインは `on_event` メソッドを通じて、システム内を流れる `VersEvent` を購読・発行します。
- **仕組み**: Kernel が「掲示板」に `ThoughtRequested` イベントを投げ、プラグインがそれに反応して `ThoughtResponse` を投げ返します。
- **メリット**: プラグイン同士が互いの存在を知らなくても連携できる「完全な疎結合」を実現します。
- **用途**: ログ監視、マルチエージェントの連鎖、バックグラウンド処理。

### ③ 能力注入による安全なリソース利用 (Capability Injection)
プラグインは自身でネットワークやファイルを操作せず、Kernel から提供された「道具（Capability）」を使用します。
- **仕組み**: `on_plugin_init` 時に Kernel から `SafeHttpClient` 等が渡されます。
- **メリット**: Kernel が通信内容を監視・制限でき、セキュリティ原則（Principle 5）を物理的に強制できます。
- **用途**: API 呼び出し、データベースアクセス、ハードウェア操作。

---

## 2. 動作シーケンス（DeepSeek プラグインの例）

1.  **初期化**: Kernel が起動し、DeepSeek プラグインをロード。`on_plugin_init` で「安全なブラウザ」を注入。
2.  **要求**: ユーザーが発言。Kernel（`MessageRouter`）が `ThoughtRequested` イベントを発行。
3.  **購読**: DeepSeek プラグインの `on_event` がイベントを検知。
4.  **実行**: プラグインが「注入されたブラウザ」を使い、DeepSeek API へリクエスト。
5.  **返信**: API からの結果を `ThoughtResponse` イベントとしてバスに流す。
6.  **完了**: Kernel がイベントを拾い、画面表示とメモリ保存を実行。

## 3. Python ブリッジとの比較

| 特徴 | Rust ネイティブ・プラグイン | Python ブリッジ・プラグイン |
| :--- | :--- | :--- |
| **通信形式** | メモリ上の直接参照 / イベント | 標準入出力を通じた JSON-RPC |
| **実行速度** | 極めて高速（オーバーヘッドなし） | 普通（シリアライズの遅延あり） |
| **セキュリティ** | Rust の型安全 + 能力注入 | プロセス分離 + 能力注入 |
| **開発の容易さ** | コンパイルが必要 | 保存するだけで即時反映 |

---

## 4. 開発指針

Rust プラグインを開発する際は、**「直接呼び出し」よりも「イベント駆動」を優先**してください。これにより、将来的に他のプラグイン（例：検閲プラグインや翻訳プラグイン）が、あなたのプラグインの動作に干渉したり協力したりすることが容易になります。
